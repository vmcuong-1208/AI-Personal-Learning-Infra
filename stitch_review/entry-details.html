Today I learned how BullMQ handles background jobs

calendar_today October 24, 2023
schedule 12 min read
Completed
ORIGINAL NOTES

BullMQ is built on top of Redis and implements a robust queuing system for Node.js. Unlike simple message brokers, it handles job persistence, retries, and complex job relationships like parent-child dependencies.

The Concurrency Model

Concurrency is defined per worker. If you set concurrency to 5, the worker will process up to 5 jobs simultaneously using the same Node.js event loop. This is critical for I/O bound tasks.

One interesting thing I found was the distinction between delayed and waiting states. A delayed job stays in a special ZSET in Redis until its timestamp is reached, whereas waiting jobs are in a standard LIST ready to be popped by workers.

Challenges encountered:

  • Understanding how stalled jobs are handled via the lock mechanism.
  • Configuring the backoff strategy properly to avoid overloading the database during retries.
  • Memory overhead of large job payloads in Redis.

Next steps: Implement a sandbox environment to test horizontal scaling of workers and observe how BullMQ redistributes jobs when a worker node fails.
